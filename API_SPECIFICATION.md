# G-ERP — API Specification

_Date: 2026-06-29_
_Status: Phase 1 design document. Derived from the frozen [PROJECT_ANALYSIS.md](PROJECT_ANALYSIS.md) architecture — API-first, every module owns its namespace, web and future mobile share the identical API surface._

This specification defines the conventions every module's API must follow, plus the full contract for the **Foundation, CRM, and Sales** endpoints built in Phase 1. Later modules (Inventory, Purchasing, Production, Installation, Finance, Reports, Marketing, AI) will add their own endpoint sets under this same contract when their phase begins.

---

## 1. Versioning

- All endpoints are prefixed `/api/v1/...`. The version is a URL path segment, not a header, for discoverability and simple routing in FastAPI.
- A breaking change to any endpoint's request/response shape requires a new version prefix (`/api/v2/...`) for that module's affected routes — additive, backward-compatible changes (new optional fields, new endpoints) do not require a version bump.
- Each module mounts under `/api/v1/<module_name>/...`, e.g.:
  - `/api/v1/crm/contacts`
  - `/api/v1/crm/deals`
  - `/api/v1/sales/quotes`
  - `/api/v1/sales/orders`
- OpenAPI schema is auto-generated by FastAPI (`/api/v1/openapi.json`, interactive docs at `/api/v1/docs`) and treated as the canonical, always-current API reference — this document describes conventions and the Phase 1 contract, not a hand-maintained duplicate of the schema.

## 2. Authentication

- **Mechanism**: JWT bearer tokens. `Authorization: Bearer <access_token>` header on every authenticated request. No server-side session/cookie-only auth path — required so mobile clients (frontend-agnostic per the frozen architecture) work identically to web.
- **Token types**:
  - **Access token**: short-lived (15 minutes), contains `user_id`, `active_company_id`, `role`, `module_permissions` claims.
  - **Refresh token**: long-lived (30 days), stored httpOnly+secure cookie for web, secure device storage for mobile; used only against `/api/v1/auth/refresh`.
- **Company context**: a user authenticated across multiple companies must select an active company; `active_company_id` is embedded in the access token after selection via `/api/v1/auth/select-company`. Every backend query derives `company_id` scoping from this claim — **never** from a client-supplied query/body parameter. Endpoints that accept a `company_id` in the request body/query are rejected if it doesn't match the token's `active_company_id` (HTTP 403).
- **Password handling**: bcrypt/argon2 hashing; never returned in any response.

### 2.1 Auth Endpoints

| Method | Path | Description |
|---|---|---|
| POST | `/api/v1/auth/login` | Body: `{email, password}` → `{access_token, refresh_token, companies: [{id, name, role}]}` |
| POST | `/api/v1/auth/select-company` | Body: `{company_id}` (with valid access token) → new `{access_token}` scoped to that company |
| POST | `/api/v1/auth/refresh` | Body: `{refresh_token}` → `{access_token}` |
| POST | `/api/v1/auth/logout` | Invalidates refresh token |
| GET | `/api/v1/auth/me` | Returns current user profile + active company + role + permissions |

## 3. Authorization (RBAC)

- Every endpoint declares the permission(s) it requires (e.g., `crm:deals:write`), enforced via a FastAPI dependency that reads the access token's `role`/`module_permissions` claims — never trusts frontend-side hiding of UI elements as the actual control.
- Roles: `owner`, `manager`, `rep`, `viewer` per company, with optional per-module overrides (per [DATABASE_DESIGN.md](DATABASE_DESIGN.md) §3.3 `user_company_roles.module_permissions`).
- Insufficient permission → `403 Forbidden` with the standard error format (§6).
- Cross-company access attempts (token's company doesn't match the resource's `company_id`) → `404 Not Found` (not 403), to avoid confirming the existence of another company's resource to an unauthorized caller.

## 4. General Conventions

- **Content type**: `application/json` for all requests/responses; file uploads use `multipart/form-data` against dedicated upload endpoints that return a `document_id` for use in other payloads.
- **Pagination**: cursor-based for all list endpoints — `?limit=25&cursor=<opaque_cursor>`. Response includes `{items: [...], next_cursor: string | null}`. Chosen over offset pagination for stability under concurrent writes (relevant once multiple modules write fast-changing data like stock movements).
- **Filtering/sorting**: list endpoints accept `?filter[field]=value` and `?sort=field` / `?sort=-field` (descending) conventions, consistently named across all modules' list endpoints.
- **Field naming**: `snake_case` in JSON payloads (matches Python/Pydantic naming, avoids a translation layer at the API boundary; frontend TypeScript types are generated from the OpenAPI schema rather than hand-written, to stay in sync automatically).
- **Timestamps**: ISO 8601 UTC (`2026-06-29T10:00:00Z`) everywhere.
- **Money**: serialized as a JSON object `{amount: "1234.50", currency: "AZN"}` (string amount to avoid floating-point precision loss in clients), never a bare float.
- **IDs**: UUIDs as strings in JSON.
- **Idempotency**: state-changing POSTs that could plausibly be retried by a flaky mobile connection (e.g., order approval) accept an optional `Idempotency-Key` header; duplicate keys within 24h return the original response rather than re-executing.

## 5. Standard Response Envelope

Successful responses are returned as the resource/collection directly (no unnecessary wrapper), e.g.:

```json
{ "id": "...", "full_name": "...", "email": "..." }
```

List responses use the pagination envelope from §4:

```json
{ "items": [ { ... }, { ... } ], "next_cursor": "eyJvZmZzZXQiOjI1fQ==" }
```

## 6. Error Format (uniform across all modules)

All errors return a consistent shape, regardless of which module raised them:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "quantity must be greater than 0",
    "details": [
      { "field": "quantity", "issue": "must be greater than 0" }
    ],
    "request_id": "9f2e3a10-..."
  }
}
```

| HTTP Status | `error.code` | Meaning |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body/query failed schema validation |
| 401 | `UNAUTHENTICATED` | Missing/invalid/expired access token |
| 403 | `FORBIDDEN` | Authenticated but lacks permission for this action |
| 404 | `NOT_FOUND` | Resource doesn't exist, or belongs to a company the caller can't access |
| 409 | `CONFLICT` | State conflict (e.g., approving an already-approved order) |
| 422 | `BUSINESS_RULE_VIOLATION` | Domain-layer rejection (e.g., invalid pipeline stage transition) — distinct from 400 because the payload is syntactically valid but semantically rejected by the domain layer |
| 429 | `RATE_LIMITED` | Rate limit exceeded (esp. auth and AI-job-submission endpoints) |
| 500 | `INTERNAL_ERROR` | Unhandled server error; `request_id` is the correlation key for log lookup |

`request_id` is always present and ties the response to a structured backend log entry, regardless of which module produced the error — set once at the FastAPI middleware layer in the core, not duplicated per module.

```json
{
  "id": "uuid", "account_id": "uuid", "deal_id": "uuid|null", "status": "draft|sent|accepted|rejected|expired",
  "total_amount": {"amount": "string", "currency": "AZN"}, "valid_until": "date|null",
  "lines": [ {"id": "uuid", "description": "string", "quantity": "string", "unit_price": {"amount":"string","currency":"AZN"}, "line_total": {"amount":"string","currency":"AZN"}} ],
  "created_at": "datetime"

```

### Sales Orders
| Method | Path | Permission | Description |
|---|---|---|---|
| GET | `/sales/orders` | `sales:orders:read` | List (filterable by status) |
| POST | `/sales/orders` | `sales:orders:write` | Create (typically from an accepted quote) |
| GET | `/sales/orders/{id}` | `sales:orders:read` | Retrieve with lines |
| POST | `/sales/orders/{id}/approve` | `sales:orders:approve` | Approves order — **publishes `OrderApproved`** (consumed later by Inventory/Production/Finance) |
| POST | `/sales/orders/{id}/cancel` | `sales:orders:write` | Cancels (only if not yet approved) |

`sales:orders:approve` is intentionally a distinct permission from `sales:orders:write` — order approval is the trigger for downstream inventory/production/finance reactions and is typically restricted to `manager`/`owner` roles, per the RBAC granularity established in the frozen architecture.

## 11. Rate Limiting

- `/api/v1/auth/*`: 10 requests/minute per IP for `login` specifically, to mitigate brute force.
- AI-job-submission endpoints (future AI module): rate-limited per company to protect shared Celery worker capacity.
- Standard endpoints: generous default (e.g., 300 requests/minute per authenticated user), tightened later if abuse patterns emerge.

## 12. Mobile Parity Notes

- No endpoint in this specification — present or future — may have a web-only equivalent that bypasses it. Any feature exposed in the Next.js frontend must be backed by one of these documented API endpoints, callable identically by a future mobile client with the same bearer token mechanism.
- Long-running operations (AI jobs, reports generation) follow an async pattern: `POST` returns `202 Accepted` with a `{job_id, status_url}`; client polls `GET status_url` or (future) receives a push notification — this pattern is mobile-friendly by design (no long-held HTTP connections).
