# G-ERP — API Specification

_Originally a Phase 1 design document (2026-06-29); rewritten 2026-07-21 against the actual current source code (every FastAPI router in `backend/`, not the original design) to close `PROJECT_AUDIT.md` Priority #4 — the previous revision stopped documenting real endpoints after Version 2.12 and still described a superseded Phase-1 CRM/Sales design that was never actually built that way._
_Status: covers all thirteen installed modules (Core, CRM, Stone Catalog, Sales, Orders, Production, Installation, Finance, Communication Center + Real Integrations, AI Sales Assistant, Reports, Purchasing, Marketing, Customer Portal) as implemented as of Version 2.33.0._

This specification defines the conventions every module's API follows, plus the full, current contract for every endpoint in the system. OpenAPI schema is auto-generated by FastAPI (`/api/v1/openapi.json`, interactive docs at `/api/v1/docs`) and is the always-current source of exact request/response shapes; this document is the human-readable map of what exists and why, verified against the router/schema source files, not hand-maintained from memory.

---

## 1. Versioning

- All endpoints are prefixed `/api/v1/...`. The version is a URL path segment, not a header, for discoverability and simple routing in FastAPI.
- Each module mounts under `/api/v1/<module_name>/...` per its `MODULE_MANIFEST` (`core/module_registry/registry.py`): `app.include_router(manifest.router, prefix=f"/api/v1/{manifest.name}")`. Module names today: `crm`, `catalog`, `sales`, `orders`, `production`, `installation`, `finance`, `communication`, `ai`, `reports`. Core's own endpoints (auth, companies, documents, events) mount directly under `/api/v1/core/...` or `/api/v1/auth/...`, outside the module-manifest system.
- A breaking change to any endpoint's request/response shape would require a new version prefix (`/api/v2/...`) for that module's affected routes — additive, backward-compatible changes (new optional fields, new endpoints) do not. No `/api/v2/...` route exists anywhere yet.

## 2. Authentication

- **Mechanism**: JWT bearer tokens. `Authorization: Bearer <access_token>` header on every authenticated request. No server-side session/cookie-only auth path — mobile clients would work identically to web.
- **Token types**:
  - **Access token**: short-lived (15 minutes), carries `user_id`, `active_company_id`, `role`, `module_permissions` claims.
  - **Refresh token**: long-lived (30 days), stamped with a `gen` (generation) claim used for revocation (see `/auth/logout` below). Stored client-side (web: `localStorage`, per `frontend/lib/session.ts` — see `PROJECT_AUDIT.md` §7/§9 for the accepted-tech-debt XSS exposure this implies).
- **Company context**: a user authenticated across multiple companies selects an active company via `/api/v1/auth/select-company`; `active_company_id` is embedded in the resulting access token. Every backend query derives `company_id` scoping from this claim — never from a client-supplied query/body parameter.
- **Password handling**: bcrypt hashing (`passlib`); never returned in any response.

### 2.1 Auth Endpoints (`core/auth/router.py`)

| Method | Path | Description |
|---|---|---|
| POST | `/api/v1/auth/login` | Body: `{email, password}` → `{access_token, refresh_token, companies: [{id, name, role}]}`. Rate-limited to 10 requests/minute per IP (§21). |
| POST | `/api/v1/auth/select-company` | Body: `{company_id}` (with a valid access token) → a new `{access_token}` scoped to that company |
| POST | `/api/v1/auth/refresh` | Body: `{refresh_token}` → `{access_token}` — rejected (401) if the token's `gen` claim is older than the user's current generation counter (see logout below) |
| POST | `/api/v1/auth/logout` | Body: `{refresh_token}` → bumps the user's generation counter, revoking **every** refresh token ever issued to them ("logout everywhere"), not just this one. Backed by `core/auth/token_denylist.py`: a per-user counter in Redis when reachable, with an automatic in-process fallback if Redis is down — not a database table. The caller's still-valid access token keeps working until its own 15-minute expiry; only future `/refresh` calls are affected. |
| GET | `/api/v1/auth/me` | Returns current user profile + active company + role + permissions |

## 3. Authorization (RBAC)

- Every endpoint declares the permission(s) it requires (e.g., `crm:customers:write`), enforced via the `require_permission(...)` FastAPI dependency (`core/rbac/dependencies.py`), which reads the access token's `role`/`module_permissions` claims — the frontend never enforces this on its own (`PROJECT_AUDIT.md` confirms no frontend-only gating exists).
- Roles: `owner`, `manager`, `rep`, `viewer` per company. Permission strings follow an action-suffix convention (`core/rbac/permissions.py`): `:read` resolves to `viewer` tier and above, `:write`/`:approve`/`:settings:write` etc. resolve to `rep` tier and above (tightened per-permission where the code says so, e.g. Finance payments).
- Insufficient permission → `403 Forbidden` (§6). A handful of endpoints intentionally bypass `require_permission` for a documented reason — every one is called out explicitly in this document where it applies (the inbound webhook receivers in §18, the events-log endpoint in §9, the document-download endpoint in §8).
- Cross-company access attempts (token's company doesn't match the resource's `company_id`) → `404 Not Found` (not 403), to avoid confirming another company's resource exists.

## 4. General Conventions

- **Content type**: `application/json` for all requests/responses; file uploads use `multipart/form-data` against the dedicated upload endpoint (§8), which returns a `document_id` for use in other payloads.
- **Pagination**: cursor-based — `?limit=25&cursor=<opaque_cursor>` (`core/api/pagination.py`: an opaque base64 offset, decoded server-side). Response is `{items: [...], next_cursor: string | null}`. Applied consistently across every list endpoint in every module (Orders/Production/Installation/Finance were fixed in Version 2.30.0 — they previously accepted `limit`/`cursor` but hardcoded `next_cursor: null`, a gap this document itself flagged during the 2.29.0 sync pass and which was closed the same day).
- **Filtering/sorting**: list endpoints accept module-specific query filters (documented per endpoint below) and `?sort=field` / `?sort=-field` (descending) where sorting is supported.
- **Field naming**: `snake_case` in JSON payloads, matching Pydantic/Python naming.
- **Timestamps**: ISO 8601 UTC. Several modules store business dates (e.g. `scheduled_date`, `expense_date`, `valid_until`) as plain `String(10)` ISO date strings rather than a `Date` column — a pragmatic SQLite/Postgres portability choice, not an oversight; see `DATABASE_DESIGN.md` for which columns use which representation.
- **Money**: monetary fields are stored and returned as `Numeric(14,2)` decimal strings (e.g. `"1234.50"`) paired with a separate `currency` field (`"AZN"`), never a bare float — avoids floating-point precision loss.
- **IDs**: UUIDs as strings in JSON.
- **Idempotency**: no `Idempotency-Key` mechanism is implemented anywhere in the current codebase (this was a Phase-1 design intention, not built) — retried POSTs are not deduplicated. Worth noting as a real, currently-open gap rather than removing the concern outright.

## 5. Standard Response Envelope

Successful single-resource responses are the resource directly:

```json
{ "id": "...", "name": "...", "status": "..." }
```

List responses use the pagination envelope from §4:

```json
{ "items": [ { ... }, { ... } ], "next_cursor": "eyJvZmZzZXQiOjI1fQ==" }
```

## 6. Error Format (uniform across all modules)

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "quantity must be greater than 0",
    "details": [ { "field": "quantity", "issue": "must be greater than 0" } ],
    "request_id": "9f2e3a10-..."
  }
}
```

| HTTP Status | `error.code` | Meaning |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body/query failed schema or business-rule validation |
| 401 | `UNAUTHENTICATED` | Missing/invalid/expired access token, or a revoked refresh token |
| 403 | `FORBIDDEN` | Authenticated but lacks permission for this action |
| 404 | `NOT_FOUND` | Resource doesn't exist, or belongs to a company the caller can't access |
| 409 | `CONFLICT` | State conflict (e.g. approving an already-approved order, a duplicate slab number) |
| 422 | Domain-specific (e.g. business-rule violation) | Syntactically valid payload rejected by the domain layer (e.g. an illegal status transition) |
| 429 | `RATE_LIMITED` | Rate limit exceeded (login only, today — see §21) |
| 500 | `INTERNAL_ERROR` | Unhandled server error; `request_id` is the correlation key for log lookup |

`request_id` is always present, set once at the FastAPI middleware layer in the core, not duplicated per module.

## 8. File Upload Endpoints (`core/storage/`, shared by every module)

| Method | Path | Permission | Description |
|---|---|---|---|
| POST | `/api/v1/core/documents` | `core:documents:write` (a single fixed permission — does **not** vary by `module`/`related_entity_type` in the body, despite that being the original Phase-1 intention) | Multipart upload (`module`, `related_entity_type`, `related_entity_id`, `file`) → stores the file (local disk in dev, Supabase Storage in production) and returns `{document_id, storage_path}` for attaching to any module's entity |
| GET | `/api/v1/core/documents/{id}` | *(authenticated only — any user in the same company can fetch the signed URL; there is no per-entity permission check, only a `document.company_id != current_user.active_company_id` ownership check → 404 on mismatch)* | Returns a signed, time-limited URL to the file |

**Upload constraints** (both added post-Phase-1, in a production-readiness pass): a 10 MB size cap (`MAX_UPLOAD_SIZE_BYTES`, → `400 VALIDATION_ERROR` if exceeded), and a content-type allowlist (`ALLOWED_UPLOAD_CONTENT_TYPES` — images, PDF, common office formats, plain text/CSV, and the audio/video types Communication Center attachments need), plus a narrow `application/octet-stream` carve-out gated on filename extension for DWG/DXF CAD files (browsers commonly report CAD files with no registered MIME type). Filenames are sanitized (`core/storage/client.py`'s `sanitize_filename()`) before ever being used to build a storage key, closing a path-traversal vector found and fixed in an earlier audit pass.

## 9. Event Log Endpoint (`core/events/`, introspection only)

| Method | Path | Access | Description |
|---|---|---|---|
| GET | `/api/v1/core/events?event_name=` | Owner role only — a hardcoded `if current_user.role != "owner": raise ForbiddenError(...)` check, **not** a registered `require_permission(...)` permission string | Read-only view into `event_log` for debugging "why did X happen"; hardcoded to the caller's active company and the most recent 100 rows (no `limit`/`cursor` pagination, and no `company_id` query param — company scoping is always implicit from the token, consistent with the rest of the API) |

## 12. Sales Module Endpoints (`/api/v1/sales/...`)

The real Sales module is a deep nested resource tree, not a flat `quotes`/`sales_orders` pair. It has two parallel structures that sound similar but serve different purposes — **don't confuse them**:
- **Quotes** (a Project's pricing document): `Quote → QuoteSection → QuoteSectionItem` / `QuoteSectionMeasurement` — everything needed to price and PDF a customer-facing quote.
- **Project workspace** (the physical execution plan, independent of any specific Quote version): `Project → Room → ProjectItem → ProjectItemMeasurement` / `ProjectItemDrawing` / `ProjectItemPhoto` — tracks the actual pieces being fabricated, their real on-site measurements, drawings, and photos, across the life of the Project regardless of how many times the Quote gets re-priced.

An Order (§13) is created from an **accepted** Quote and deep-copies its Sections/Items/Measurements at that moment — editing an Order afterward never touches the source Quote.

Provider-agnostic AI recommendations layered on CRM, Communication, Sales, and Tasks. **The AI module never performs a business action automatically** — every analysis call only creates `AIRecommendation` rows; a recommendation only ever affects Lead/Customer/Conversation/Task/Quote data if a human explicitly accepts it through the review endpoint, and even then that endpoint only ever updates the recommendation's own status. **No real LLM provider is integrated yet, by explicit design** — every provider name (`mock`, `openai`, `anthropic`, `gemini`, `ollama`, `azure_openai`) currently resolves to `MockAIProvider`, a deterministic heuristic engine.

| Method | Path | Permission | Description |
|---|---|---|---|
| POST | `/ai/leads/{lead_id}/analyze` | `ai:recommendations:write` | CRM Intelligence on a Lead — `lead_score`, `win_probability`, `priority_recommendation`, `next_best_action`, `follow_up_recommendation`, `lead_quality_explanation`, and conditionally `duplicate_lead`/`similar_customer`/`missing_info` |
| POST | `/ai/conversations/{id}/analyze` | `ai:recommendations:write` | Communication Intelligence on a Conversation's transcript — language/intent/sentiment/urgency, a summary, and conditionally entity extraction / link suggestions |
| POST | `/ai/quotes/{id}/analyze` | `ai:recommendations:write` | Sales Intelligence on a Quote against this company's own historical accepted quotes — discount guidance, delivery-complexity estimate, and conditionally product/cross-sell/upsell recommendations, margin-risk and price-anomaly detection |
| POST | `/ai/tasks/suggest` | `ai:recommendations:write` | Task Intelligence company-wide — task/reminder suggestions for stale leads and unanswered conversations, assignee suggestions, overdue-risk flags |
| GET | `/ai/recommendations?analysis_kind=&recommendation_type=&related_entity_type=&related_entity_id=&status=&provider=&sort=&limit=&cursor=` | `ai:recommendations:read` | List, cursor-paginated |
| GET | `/ai/recommendations/{id}` | `ai:recommendations:read` | Get one, including full audit fields (`provider`, `model`, `prompt`, `response`, `confidence_score`, `execution_time_ms`, `requested_by`) |
| POST | `/ai/recommendations/{id}/review` | `ai:recommendations:write` | Body `{decision: "accept"\|"reject"\|"edit", edited_response?}` — `422` if already reviewed (a decision is final) |
| GET | `/ai/dashboard` | `ai:dashboard:read` | Aggregated AI Dashboard: lead score distribution, pipeline health, at-risk customers, today's recommendations, usage statistics |

All endpoints publish a corresponding analysis-complete event (`LeadAnalyzed`, `ConversationAnalyzed`, `QuoteAnalyzed`, `TasksSuggested`) or review-decision event (`RecommendationAccepted`/`Rejected`/`Edited`), per the cross-cutting audit+event rule every module follows.

## 21. Marketing Module Endpoints (`/api/v1/marketing/...`, Version 2.32.0)

Marketing campaigns with real lead attribution and revenue performance. `depends_on=["crm", "orders"]` — read-only: the performance endpoint queries `crm_leads` (filtered by the campaign's id) and `orders` (for converted leads' customers) directly, the same "depends_on for read access" pattern Reports uses against every other module, not a write-side coupling. ## 22. Customer Portal Module Endpoints (`/api/v1/customer_portal/...`, Version 2.33.0)

A second, entirely separate authentication identity from everything above: a customer of G-STONE (not a staff member) can log in and see their own orders, quotes, invoices, installation schedule, and documents. `depends_on=["crm", "sales", "orders", "finance", "installation"]` — `CustomerLogin` (`customer_portal_logins`) carries a real FK to `crm_customers.id` (safe here, unlike Marketing's `campaign_id`, because this table lives in the dependent module, not in CRM), and the read endpoints below query Order/Quote/Invoice/InstallationJob directly, the same "depends_on for read access" pattern Reports and Marketing use.

**A customer token is never a staff token and vice versa.** Staff tokens (§2) carry `"type": "access"`/`"refresh"` and a `role`/`module_permissions` shape; customer tokens carry `"type": "customer_access"`/`"customer_refresh"` and only `customer_id`/`company_id` — no role, no permissions, because a customer isn't a member of the RBAC hierarchy at all. The two are cryptographically indistinguishable except for that `type` claim, which both `core.rbac.dependencies.get_current_user` and `customer_portal`'s own `get_current_customer` dependency check and reject a mismatch on — confirmed by both automated tests and a live end-to-end smoke test against the dev database.

### Admin (staff-facing) access management
| Method | Path | Permission | Description |
|---|---|---|---|
| GET | `/customer_portal/admin/customers/{customer_id}/access` | `customer_portal:access:read` | Get the portal login state for a customer — `404` if portal access was never enabled |
| POST | `/customer_portal/admin/customers/{customer_id}/access` | `customer_portal:access:write` | Enable portal access. Body: `{email, password}` — `409` if this customer already has portal access, or if the email is already used by another portal login (portal emails are globally unique, same as staff `users.email`) |
| POST | `/customer_portal/admin/customers/{customer_id}/access/reset-password` | `customer_portal:access:write` | Set a new password. Body: `{password}` — `404` if access was never enabled |
| POST | `/customer_portal/admin/customers/{customer_id}/access/status` | `customer_portal:access:write` | Enable/disable login without discarding the account. Body: `{is_active}` — `404` if access was never enabled |

## 23. Cut Optimization Module Endpoints (`/api/v1/cut_optimization/...`, Version 2.35.0 — Stone Fabrication Workflow Phase 2)

Slab cutting/nesting optimization and Smart Offcut Management, built on top of Phase 1's slab lifecycle and offcut tracking. `depends_on=["catalog"]` — reads Slab/Material directly for dimensions and offcut candidates, the same "module → core/other-module, never reversed" direction every module in this codebase already follows. Owns one table, `cut_optimization_runs` (Optimization History — see `DATABASE_DESIGN.md` §16 for the full schema). Also see `DATABASE_DESIGN.md` §5.6 for the Offcut Library note on the new `GET /catalog/slabs/offcuts` search endpoint (Catalog module, listed in §11 above) and §8.6 for the Production Planning Dashboard (`GET /reports/production-planning`, Reports module, §17 above).

| Method | Path | Permission | Description |
|---|---|---|---|
| POST | `/cut_optimization/runs` | `cut_optimization:write` | Run the nesting algorithm. Body: `{pieces: [{label, length_mm, width_mm, quantity, allow_rotation}], kerf_mm, slab_id?, slab_length_mm?, slab_width_mm?, material_id?, notes?}` — either `slab_id` (a real Catalog slab/offcut) or both `slab_length_mm`/`slab_width_mm` (a hypothetical slab) is required, `422` otherwise. Every run is persisted (Optimization History) and returned with `placements` (x/y/length/width/rotated per piece instance) and `unplaced` (pieces that didn't fit, with a reason) |
| GET | `/cut_optimization/runs?material_id=&slab_id=&limit=&cursor=` | `cut_optimization:read` | List history, cursor-paginated |
| GET | `/cut_optimization/runs/{id}` | `cut_optimization:read` | Reopen a previous layout — the run is an immutable snapshot, nothing is recomputed |
| POST | `/cut_optimization/recommendations` | `cut_optimization:read` | Smart Offcut Management. Body: `{material_id, pieces, kerf_mm, thickness_mm?, finish?, warehouse_id?, persist_top_candidate?}`. Searches every `available` offcut matching material/thickness/finish, runs the same nesting algorithm against each as a candidate slab, and ranks the ones that fit by utilization descending (least waste first, tie-broken by the smaller candidate). Returns `recommend_new_slab: true` with a reason only when no offcut in stock can fit the requirement. The winning candidate is persisted as a `source: "offcut_recommendation"` Optimization History run by default (`persist_top_candidate`, default `true`) |

## 24. Rate Limiting

- `POST /api/v1/auth/login`: 10 requests/minute per IP (`core/rbac/rate_limit.py`, `FixedWindowRateLimiter`), confirmed accurate against the current code. In-process only today — a multi-instance deployment would need a shared store (Redis, already provisioned elsewhere in this stack) instead; documented as the known upgrade path in the code itself.
- `POST /api/v1/customer_portal/auth/login`: 10 requests/minute per IP, its own `FixedWindowRateLimiter` instance (`modules/customer_portal/presentation/api/auth.py`) — deliberately a separate bucket from staff login so a burst of customer login attempts from one IP can never throttle a staff member on the same network, and vice versa.
- No other endpoint in this API is rate-limited today. The Phase-1 design intention of a generous default rate limit across all authenticated endpoints, and per-company limits on AI-job submission, were never implemented — noted here as an open item, not a false claim of current behavior.

## 25. Mobile Parity Notes

- No endpoint in this specification has a web-only equivalent that bypasses it — every feature in the Next.js frontend is backed by one of the documented endpoints above, callable identically by a future mobile client with the same bearer-token mechanism. This structural property is real (verified during `PROJECT_AUDIT.md`'s backend review: no server-rendered-only or session-cookie-only path exists), even though no mobile client has been built yet to actually exercise it.
- **Correction from the original Phase-1 draft of this document**: no endpoint anywhere in the current codebase uses the `202 Accepted` + `{job_id, status_url}` polling pattern originally proposed for long-running operations (AI analysis, report export). Both are synchronous today — AI analysis because `MockAIProvider` is fast deterministic heuristics, not a real model call; report export because it's a direct database aggregation, not a queued job. Should a real LLM provider or a genuinely slow export ever change that, the async pattern remains the documented target design, but it is not current behavior.
