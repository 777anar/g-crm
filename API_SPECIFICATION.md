# G-ERP ŌĆö API Specification

_Date: 2026-06-29_
_Status: Phase 1 design document. Derived from the frozen [PROJECT_ANALYSIS.md](PROJECT_ANALYSIS.md) architecture ŌĆö API-first, every module owns its namespace, web and future mobile share the identical API surface._

This specification defines the conventions every module's API must follow, plus the full contract for the **Foundation, CRM, and Sales** endpoints built in Phase 1. Later modules (Inventory, Purchasing, Production, Installation, Finance, Reports, Marketing, AI) will add their own endpoint sets under this same contract when their phase begins.

---

## 1. Versioning

- All endpoints are prefixed `/api/v1/...`. The version is a URL path segment, not a header, for discoverability and simple routing in FastAPI.
- A breaking change to any endpoint's request/response shape requires a new version prefix (`/api/v2/...`) for that module's affected routes ŌĆö additive, backward-compatible changes (new optional fields, new endpoints) do not require a version bump.
- Each module mounts under `/api/v1/<module_name>/...`, e.g.:
  - `/api/v1/crm/contacts`
  - `/api/v1/crm/deals`
  - `/api/v1/sales/quotes`
  - `/api/v1/sales/orders`
- OpenAPI schema is auto-generated by FastAPI (`/api/v1/openapi.json`, interactive docs at `/api/v1/docs`) and treated as the canonical, always-current API reference ŌĆö this document describes conventions and the Phase 1 contract, not a hand-maintained duplicate of the schema.

## 2. Authentication

- **Mechanism**: JWT bearer tokens. `Authorization: Bearer <access_token>` header on every authenticated request. No server-side session/cookie-only auth path ŌĆö required so mobile clients (frontend-agnostic per the frozen architecture) work identically to web.
- **Token types**:
  - **Access token**: short-lived (15 minutes), contains `user_id`, `active_company_id`, `role`, `module_permissions` claims.
  - **Refresh token**: long-lived (30 days), stored httpOnly+secure cookie for web, secure device storage for mobile; used only against `/api/v1/auth/refresh`.
- **Company context**: a user authenticated across multiple companies must select an active company; `active_company_id` is embedded in the access token after selection via `/api/v1/auth/select-company`. Every backend query derives `company_id` scoping from this claim ŌĆö **never** from a client-supplied query/body parameter. Endpoints that accept a `company_id` in the request body/query are rejected if it doesn't match the token's `active_company_id` (HTTP 403).
- **Password handling**: bcrypt/argon2 hashing; never returned in any response.

## 3. Authorization (RBAC)

- Every endpoint declares the permission(s) it requires (e.g., `crm:deals:write`), enforced via a FastAPI dependency that reads the access token's `role`/`module_permissions` claims ŌĆö never trusts frontend-side hiding of UI elements as the actual control.
- Roles: `owner`, `manager`, `rep`, `viewer` per company, with optional per-module overrides (per [DATABASE_DESIGN.md](DATABASE_DESIGN.md) ┬¦3.3 `user_company_roles.module_permissions`).
- Insufficient permission ŌåÆ `403 Forbidden` with the standard error format (┬¦6).
- Cross-company access attempts (token's company doesn't match the resource's `company_id`) ŌåÆ `404 Not Found` (not 403), to avoid confirming the existence of another company's resource to an unauthorized caller.

## 4. General Conventions

- **Content type**: `application/json` for all requests/responses; file uploads use `multipart/form-data` against dedicated upload endpoints that return a `document_id` for use in other payloads.
- **Pagination**: cursor-based for all list endpoints ŌĆö `?limit=25&cursor=<opaque_cursor>`. Response includes `{items: [...], next_cursor: string | null}`. Chosen over offset pagination for stability under concurrent writes (relevant once multiple modules write fast-changing data like stock movements).
- **Filtering/sorting**: list endpoints accept `?filter[field]=value` and `?sort=field` / `?sort=-field` (descending) conventions, consistently named across all modules' list endpoints.
- **Field naming**: `snake_case` in JSON payloads (matches Python/Pydantic naming, avoids a translation layer at the API boundary; frontend TypeScript types are generated from the OpenAPI schema rather than hand-written, to stay in sync automatically).
- **Timestamps**: ISO 8601 UTC (`2026-06-29T10:00:00Z`) everywhere.
- **Money**: serialized as a JSON object `{amount: "1234.50", currency: "AZN"}` (string amount to avoid floating-point precision loss in clients), never a bare float.
- **IDs**: UUIDs as strings in JSON.
- **Idempotency**: state-changing POSTs that could plausibly be retried by a flaky mobile connection (e.g., order approval) accept an optional `Idempotency-Key` header; duplicate keys within 24h return the original response rather than re-executing.

## 5. Standard Response Envelope

Successful responses are returned as the resource/collection directly (no unnecessary wrapper), e.g.:

```json
{ "id": "...", "full_name": "...", "email": "..." }
```

List responses use the pagination envelope from ┬¦4:

```json
{ "items": [ { ... }, { ... } ], "next_cursor": "eyJvZmZzZXQiOjI1fQ==" }
```

## 6. Error Format (uniform across all modules)

All errors return a consistent shape, regardless of which module raised them:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "quantity must be greater than 0",
    "details": [
      { "field": "quantity", "issue": "must be greater than 0" }
    ],
    "request_id": "9f2e3a10-..."
  }
}
```

| HTTP Status | `error.code` | Meaning |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Request body/query failed schema validation |
| 401 | `UNAUTHENTICATED` | Missing/invalid/expired access token |
| 403 | `FORBIDDEN` | Authenticated but lacks permission for this action |
| 404 | `NOT_FOUND` | Resource doesn't exist, or belongs to a company the caller can't access |
| 409 | `CONFLICT` | State conflict (e.g., approving an already-approved order) |
| 422 | `BUSINESS_RULE_VIOLATION` | Domain-layer rejection (e.g., invalid pipeline stage transition) ŌĆö distinct from 400 because the payload is syntactically valid but semantically rejected by the domain layer |
| 429 | `RATE_LIMITED` | Rate limit exceeded (esp. auth and AI-job-submission endpoints) |
| 500 | `INTERNAL_ERROR` | Unhandled server error; `request_id` is the correlation key for log lookup |

`request_id` is always present and ties the response to a structured backend log entry, regardless of which module produced the error ŌĆö set once at the FastAPI middleware layer in the core, not duplicated per module.

## 11. Rate Limiting

- `/api/v1/auth/*`: 10 requests/minute per IP for `login` specifically, to mitigate brute force.
- AI-job-submission endpoints (future AI module): rate-limited per company to protect shared Celery worker capacity.
- Standard endpoints: generous default (e.g., 300 requests/minute per authenticated user), tightened later if abuse patterns emerge.

## 12. Mobile Parity Notes

- No endpoint in this specification ŌĆö present or future ŌĆö may have a web-only equivalent that bypasses it. Any feature exposed in the Next.js frontend must be backed by one of these documented API endpoints, callable identically by a future mobile client with the same bearer token mechanism.
- Long-running operations (AI jobs, reports generation) follow an async pattern: `POST` returns `202 Accepted` with a `{job_id, status_url}`; client polls `GET status_url` or (future) receives a push notification ŌĆö this pattern is mobile-friendly by design (no long-held HTTP connections).

## 14. AI Sales Assistant Module Endpoints (`/api/v1/ai/...`, Version 2.8, as actually implemented)

Provider-agnostic AI recommendations layered on top of CRM, Communication, Sales, and Tasks. **The AI module never performs a business action automatically** ŌĆö every analysis call only ever creates `AIRecommendation` rows; a recommendation only ever affects Lead/Customer/Conversation/Task/Quote data if a human explicitly accepts it through the review endpoint below, and even then the review endpoint itself never touches another module's tables ŌĆö it only updates the recommendation's own status. This is enforced structurally, not just by convention (see `modules/ai/application/use_cases/review_use_case.py`).

**No real LLM provider is integrated yet, by explicit design.** Every provider name (`mock`, `openai`, `anthropic`, `gemini`, `ollama`, `azure_openai`) currently resolves to `MockAIProvider` (see `modules/ai/infrastructure/providers/`), a deterministic heuristic engine (regex/keyword/statistics based, not a real model call) that lets the rest of the platform ŌĆö schemas, audit trail, dashboard, frontend ŌĆö be built and tested against a stable contract today. Wiring in a real provider later is a one-line change in `infrastructure/providers/registry.py`; no endpoint, schema, or use case changes shape.

## 15. Real Integrations Endpoints (`/api/v1/communication/...`, Version 2.9, as actually implemented)

## 16. Project Workspace Endpoints (`/api/v1/sales/...`, Version 2.11, as actually implemented, extends ┬¦8)

Sprint 3 ("Project" as the primary business object): Rooms and Project Items are new sub-resources of an existing Project, independent of any Quote version. All endpoints below use the existing `sales:projects:read`/`sales:projects:write` permissions (matching how Quote Sections/Items/Measurements already ride on `sales:quotes:read`/`write` rather than getting their own permission strings) ŌĆö no new permission strings were introduced.

| Method | Path | Description |
|---|---|---|
| GET/POST | `/sales/projects/{project_id}/rooms` | List/create Rooms ŌĆö publishes `RoomCreated` |
| PATCH/DELETE | `/sales/rooms/{room_id}` | Update/delete a Room ŌĆö deleting cascades to its Project Items (and their Measurements/Drawings/Photos) |
| GET/POST | `/sales/rooms/{room_id}/items` | List/create Project Items within a Room ŌĆö publishes `ProjectItemCreated` |
| GET | `/sales/projects/{project_id}/items` | All Project Items across every Room in a project (used by the workspace's Measurements/Drawings/Photos/Production/Installation/Completion tabs) |
| PATCH/DELETE | `/sales/project-items/{item_id}` | Update (including `production_status`/`installation_status`) or delete a Project Item |
| GET/POST | `/sales/project-items/{item_id}/measurements` | List/create measurement revisions ŌĆö every POST is a new revision (never an overwrite); publishes `ProjectItemMeasurementRecorded` |
| PATCH/DELETE | `/sales/project-item-measurements/{measurement_id}` | Edit a revision in place (e.g. attach `customer_signature_document_id` and mark `final`) or delete it |
| GET/POST | `/sales/project-items/{item_id}/drawings` | List/attach a DWG/DXF/sketch/PDF drawing (`document_id` from a prior `POST /core/documents` upload) |
| DELETE | `/sales/project-item-drawings/{drawing_id}` | Detach a drawing |
| GET/POST | `/sales/project-items/{item_id}/photos` | List/attach a site photo |
| DELETE | `/sales/project-item-photos/{photo_id}` | Detach a photo |

Drawings/Photos/the measurement signature all reuse the existing `POST /api/v1/core/documents` upload endpoint (`module=sales`, `related_entity_type` one of `project_item_drawing`\|`project_item_photo`\|`project_item_measurement`) ŌĆö the same generic Documents pipeline every other module's file attachments already use. That endpoint's content-type allowlist was extended to accept DWG/DXF files.

A rejected signature returns `403` and is still logged (`direction=inbound`, `signature_valid=false`) rather than silently dropped, since a misconfigured secret or a spoofing attempt is itself an operational signal worth surfacing in the Webhook Monitor. A processing error *after* signature verification succeeds never propagates as a 500 to the provider (which would trigger its own retry storm) ŌĆö it's caught, logged, and the endpoint still returns `200`.
